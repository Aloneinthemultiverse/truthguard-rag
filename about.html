Introducing TruthGuard

RAG that knows when it doesn't know

Standard retrieval-augmented generation answers every question with equal confidence — including the ones it should refuse. TruthGuard places an assessment gate between retrieval and generation: the generator is never invoked until the retrieved context has been checked for sufficiency, contradiction, and ambiguity. A model that isn't called on bad context cannot fabricate from it.

The problem

Four ways retrieval quietly lies

RAG failures are rarely loud. The answer looks confident, cites a source, and reads well — but the source was missing, contradicted, unreadable, or answering a different question than the one asked.

P1
Blindness

Scanned pages carry no text layer. The retriever never sees them, so the fact does not exist as far as the system is concerned.

P2
Omission

The answer isn't in the corpus at all. Rather than say so, the model composes one from adjacent context.

P3
Silent arbitration

Two documents disagree. The model picks one, presents it as fact, and never mentions the conflict existed.

P4
Ambiguity collapse

The question had two readings. The model answers one and behaves as if the other was never possible.

The design consequence: these are not prompt problems. Asking a model to "be careful" does not fix a missing text layer or a contradiction between two PDFs. They have to be caught structurally, before generation.

How it works

The generator fires last, not first

Retrieval produces candidate context. The assessment gate then runs four checks, cheapest first, and decides what kind of response is even permissible. Only a clean pass reaches the generator.

01
Ingest

Mixed PDFs, scans and code-in-PDF normalized to Markdown. Pages without a text layer escalate through an OCR ladder with per-word confidence.

02
Retrieve

Three signals scored in parallel — dense vectors, BM25 keywords, and exact entity matches — fused by reciprocal rank and reranked.

03
Assess

Sufficiency, then fact extraction with validity windows, then contradiction detection, then an answerability verdict.

04
Respond

Answer with citations and a confidence band — or refuse, clarify, or present both sides of a conflict.

Four possible outcomes

Answer

Context is sufficient and consistent. Generate with chunk-level citations and a confidence band.

Dual-answer

Sources conflict. Present both values with their provenance rather than silently choosing.

Clarify

The question is ambiguous. Offer the readings, then re-run with the user's choice.

Refuse

The corpus does not contain the answer. Decline and report what is missing.

Inside the gate

  • Sufficiency — maximum retrieval similarity against a threshold. Fails here cost zero LLM tokens.
  • Fact extraction — retrieved text becomes (subject, relation, object) triples with qualifiers and bi-temporal validity windows.
  • Contradiction detection — same subject and relation, overlapping validity, different canonical value. Non-overlapping windows are treated as a timeline, not a conflict.
  • Answerability — a verdict of sufficient, partial, insufficient, or ambiguous, with clarifying options when the question is underspecified.

Context memory

One graph for every conversation, document, and repository

Memory here is not a chat log. Conversations, documents, and codebases are compiled into a single global graph, so recall costs proportional to a neighborhood rather than to history — and the same memory is reachable from any model through MCP.

y+  KNOWLEDGE
Documents

Provenance-tagged chunks, extracted entities, and community summaries with a compiled truth per topic.

x  SPINE
Conversation

Turns stored as decision memory with confidence, decay, and supersession. Each session is its own thread.

y−  CODE
Codebase

Call and import structure alongside real function bodies, extracted across more than twenty languages.

The planes are cross-wired: a chat turn is linked to the document passage it grounded on and the code it referenced, so any claim can be traced back to its source in one hop.

Reproduction notes

What we checked, including what didn't work

We attempted a direct comparison against a competing graph-memory system by running its published pipeline on the same conversational data. Its deterministic path produced no conversational nodes, and its LLM path produced nodes with no edges, leaving its graph query unable to traverse. The published figures for that system rely on a harness not included in its repository.

We report this because the alternative — quoting a favorable number without noting that we could not reproduce the baseline — is exactly the failure mode this project exists to prevent.

On judges. The same memory system can score anywhere from 27% to 68% on the same benchmark depending on whose harness and judge model runs it. Absolute cross-paper comparison is unreliable; we publish our methodology and a baseline calibration so ours can be checked.

Built on

Deterministic where it counts

Graph construction, retrieval, and structural queries are deterministic and run locally. The language model is used for judgment — extraction, summarization, generation — not for bookkeeping.
